## Milestone 10: Client Portal

Built the full client-facing portal across four routes. All data access goes through `withRLS()` — RLS policies on the `tickets`, `ticket_messages`, and `ticket_feedback` tables enforce that clients can only read/write their own data at the DB layer, so no application-level filtering is needed beyond passing the correct context. `getMyTickets` uses `COUNT(*) OVER()` for single-query pagination (same pattern as client analysis). `getTicketDetail` runs 3 queries within one `withRLS` transaction: ticket + type join, messages + team member name join, and feedback lookup. `createTicket` inserts the ticket and first message in the same transaction; `clientId` is always sourced from the session context (`ctx.clientId`), never user input, so spoofing is impossible. `submitFeedback` relies on the `ticket_feedback` INSERT policy (inherited from Milestone 3) which verifies the ticket belongs to the authenticated client. Portal layout (`src/app/portal/layout.tsx`) is a server component that role-guards and redirects `team_member` users to `/dashboard`; the middleware provides the same guard at the Edge before the request even reaches the layout. Client components: `SignOutButton` calls `authClient.signOut()` then `router.push('/sign-in')`; `NewTicketForm` calls the `createTicket` server action and uses `router.push` to navigate to the new ticket on success; `FeedbackForm` calls `submitFeedback` and uses `router.refresh()` so the parent server component re-renders (hiding the form and showing the saved feedback) without a full navigation. Added `src/components/ui/textarea.tsx` (standard shadcn pattern, was absent from the original install). Ticket detail message thread renders client messages right-aligned with a blue background and team messages left-aligned with muted background.

## Milestone 11: Performance Optimization + Testing

Benchmarked all 8 major queries via `scripts/benchmark.ts` (3 runs each, wall-clock including Supabase network RTT). All met targets after one fix. `get_client_analysis_rls` initially ran at 348 ms — EXPLAIN ANALYZE revealed a cross-join: joining both `tickets` and `payments` directly to `clients` produced 283,940 intermediate rows (50 clients × 5 payments × 1,213 tickets avg) before GROUP BY. This also caused a data accuracy bug where `total_tickets` was inflated by the payment count per client. Fixed by rewriting the function with `ticket_stats` and `payment_stats` CTEs that pre-aggregate each table to 50 rows before joining; also replaced `EXECUTE FORMAT` dynamic SQL with static SQL + CASE-based ORDER BY for plan caching. Result: 348 ms → 135 ms. E2E testing via browser DevTools revealed that Next.js serialises concurrent server action POST requests from the same page — three pages were firing two sequential requests instead of one parallel one (Dashboard: +187 ms wasted, Distribution: +392 ms, Response Time: +260 ms). Fixed by introducing combined server actions (`getDashboardAll`, `getDistributionAll`, `getResponseTimeAll`) that use `Promise.all` for server-side parallelism and return both data sets in a single HTTP round-trip. Client components updated to a single `useQuery` each; `OverdueTicketsTable` converted from self-contained query component to a data-prop component with page state lifted to `ResponseTimeContent`. Full results documented in `docs/performance-results.md`.

## Milestone 9: Response Time Analysis

Built two RLS-aware stored functions in `database/rls-functions.sql`: `get_resolution_time_stats_rls` computes min/max/avg/median actual resolution hours grouped by ticket priority using `PERCENTILE_CONT(0.5)`, with `expected_hours` derived as the average of `ticket_types.avg_resolution_hours` for resolved tickets in each priority bucket; `get_overdue_tickets_rls` returns paginated resolved tickets where actual hours exceeded the type benchmark, ordered by `excess_hours DESC`, with `COUNT(*) OVER()` for single-query pagination. Both accept date and team member filters only — ticket type and priority are the measured dimensions so they are excluded from filtering. Server actions `getResolutionTimeStats()` and `getOverdueTickets()` in `src/lib/queries/response-time.ts` call these functions via `adminDb.execute()` in autocommit (same pattern as dashboard summary). Built `ResolutionComparisonChart` (Recharts grouped BarChart, Actual Avg in blue vs Expected in slate, Y-axis formatted with `h` suffix) and `OverdueTicketsTable` (paginated, resets to page 1 on filter change via `useEffect`, `keepPreviousData` for opacity fade). Response-time page split into server component (`page.tsx`) for role guard + client component (`ResponseTimeContent.tsx`) for all interactive logic — matching the clients page pattern. Stats table variance column is color-coded green/red with percentage delta alongside the hour delta.

## Milestone 8: Client Analysis View

Built `getClientAnalysis()` server action in `src/lib/queries/clients.ts`. Initial implementation used `withRLS()` with two sequential queries (COUNT + data), totalling 4 DB round-trips and ~650ms latency. Optimised in two stages: first merged the COUNT into the data query using `COUNT(*) OVER()` window function (3 round-trips, ~500ms); then moved all RLS setup + query into a `get_client_analysis_rls()` stored function in `database/rls-functions.sql` called via `adminDb.execute()` in autocommit (1 round-trip, ~300ms) — halving end-to-end latency. Dynamic ORDER BY is handled inside the function with `EXECUTE format()` + `USING` clause; the sort column is whitelisted via a `CASE` expression to prevent SQL injection. `last_ticket_date` is explicitly cast to ISO TEXT inside the function using `TO_CHAR(..., 'YYYY-MM-DD"T"HH24:MI:SS"Z"')` to avoid ambiguity between Date objects and strings across postgres.js versions. Built `ClientAnalysisTable` client component with debounced search (300ms, resets page), sortable column headers with ↑↓↕ indicators, plan type badges (starter=gray, professional=blue, enterprise=purple), and server-side pagination. Used `keepPreviousData` from TanStack Query to show stale data faded at 60% opacity during refetch instead of skeleton loaders, eliminating the flash on sort and page changes. Clients page at `src/app/dashboard/clients/page.tsx` is a server component that calls `getUserContext()` and redirects to `/portal` if `role !== 'team_member'` before rendering the table.

## Milestone 7: Distribution Charts

Each strips `ticketType` and `priority` from the incoming `FilterState` before applying filters, so the charts always show a complete distribution scoped only by date and team member. `getTicketsByType` JOINs `tickets → ticket_types`, groups by type, and computes percentage server-side. `getTicketsByPriority` uses conditional aggregation (`SUM(CASE WHEN status = '...' Added `allowedFilters?: FilterKey[]` prop to `FilterBar` — the distribution page passes `['date', 'teamMember']` to restrict the filter UI to relevant dimensions only. Built `TicketsByTypeChart` (Recharts PieChart donut, `innerRadius=60 outerRadius=110`, 14-color palette, percentage labels on slices ≥4%, suppressed on small slices to avoid clutter) and `TicketsByPriorityChart` (stacked BarChart, open=red/in_progress=yellow/resolved=green). ## Milestone 6: Team Performance Table

Built `getTeamPerformance()` server action in `src/lib/queries/team.ts` using `withRLS()`. Single query LEFT JOINs `team_members → tickets → ticket_feedback`, aggregating `COUNT`, `COUNT FILTER`, `AVG(EXTRACT(EPOCH ...))`, and `AVG(rating)` per team member — all 15 members returned including those with 0 tickets. Postgres returns numeric aggregates as strings via postgres.js, so all values cast through `Number()` on return. Built `TeamPerformanceTable` client component at `src/components/dashboard/TeamPerformanceTable.tsx` using `@tanstack/react-table` with `getSortedRowModel` + `getFilteredRowModel`. All columns sortable via click; text columns (Name, Department, Status) have a text input filter; numeric columns have min/max range inputs via a custom `numberRangeFilter` FilterFn. Key bug fixed: TanStack Table calls `filterFn.autoRemove` with `undefined` when clearing a filter value — destructuring `undefined` as an array throws "val is not iterable"; fixed with an `!Array.isArray(val)` guard returning `true` (remove filter). Top performer computed client-side: member with the highest resolution rate whose rating is at or above the field average; highlighted with green row tint and a "Top Performer" badge. Null values display as `—` throughout.



## Milestone 5: Dashboard Overview Cards + Tickets Over Time

Built `getDashboardSummary` and `getTicketsOverTime` server actions in `src/lib/queries/dashboard.ts`. Initial approach used `withRLS` + Drizzle ORM queries, but profiling revealed ~850ms load times caused by 5 sequential DB round-trips per query (BEGIN → SET LOCAL ROLE → set_config ×4 → query → COMMIT) at ~170ms Supabase latency each. Optimised in stages: first batched 4 `set_config` calls into one SELECT (5→2 setup round-trips); then moved RLS setup + query into Postgres stored functions (`get_dashboard_summary_rls`, `get_tickets_over_time_rls` in `database/rls-functions.sql`) called in autocommit via `adminDb.execute()`, collapsing the entire operation to 1 DB round-trip; finally enabled BetterAuth `cookieCache` (maxAge 300s) to eliminate the session validation DB round-trip, and wrapped `getUserContext` in React `cache()`. Result: initial load and filtered queries both under 300ms. Parameter serialisation required care: JavaScript `Date` objects and `number[]` arrays are incompatible with Drizzle's `sql` template in this context — fixed by converting dates to ISO strings and arrays to Postgres array literal strings (`'{1,2}'`) with explicit `::timestamptz` / `::int[]` / `::text[]` casts in the SQL. Built `DashboardContent` client component with `FilterBar`, 4 summary cards with loading skeletons, and `TicketsOverTimeChart` (Recharts `LineChart`, two series). Dashboard page at `src/app/dashboard/page.tsx` renders the client component; both TanStack queries include `filters` in their query keys for automatic refetch on filter change.

## Milestone 3: RLS Policies + Database Role Setup

Created a `rls_user` Postgres role (NOLOGIN, NOBYPASSRLS) and granted it baseline access to the public schema. Enabled RLS with FORCE on all 7 tables. Added 4 Postgres helper functions (`get_app_user_role`, `get_app_client_id`, `get_app_user_id`, `get_app_team_member_id`) that read transaction-local `set_config` variables — all without SECURITY DEFINER so they run as the calling role. Created SELECT, INSERT, and UPDATE policies: team members see all data; clients are filtered to their own `client_id`. The `messages_insert` policy enforces attribution — team member inserts must match `from_team_member_id = get_app_team_member_id()`, client inserts must have `from_team_member_id IS NULL`, preventing either party from spoofing the other. Policies verified manually in Supabase SQL Editor. Updated `src/lib/db/index.ts` to a single `adminDb` connection (removed `rlsDb` and `DATABASE_RLS_URL`). Created `src/lib/db/rls-client.ts` with `withRLS(ctx, fn)`: opens a transaction on `adminDb`, issues `SET LOCAL ROLE rls_user` as the first statement (transaction-scoped, auto-resets on commit/rollback), then sets all 4 session vars. `withRLS` accepts the full `UserContext` object from `getUserContext()`.
