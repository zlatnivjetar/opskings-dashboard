## Milestone 5: Dashboard Overview Cards + Tickets Over Time

Built `getDashboardSummary` and `getTicketsOverTime` server actions in `src/lib/queries/dashboard.ts`. Initial approach used `withRLS` + Drizzle ORM queries, but profiling revealed ~850ms load times caused by 5 sequential DB round-trips per query (BEGIN â†’ SET LOCAL ROLE â†’ set_config Ă—4 â†’ query â†’ COMMIT) at ~170ms Supabase latency each. Optimised in stages: first batched 4 `set_config` calls into one SELECT (5â†’2 setup round-trips); then moved RLS setup + query into Postgres stored functions (`get_dashboard_summary_rls`, `get_tickets_over_time_rls` in `database/rls-functions.sql`) called in autocommit via `adminDb.execute()`, collapsing the entire operation to 1 DB round-trip; finally enabled BetterAuth `cookieCache` (maxAge 300s) to eliminate the session validation DB round-trip, and wrapped `getUserContext` in React `cache()`. Result: initial load and filtered queries both under 300ms. Parameter serialisation required care: JavaScript `Date` objects and `number[]` arrays are incompatible with Drizzle's `sql` template in this context â€” fixed by converting dates to ISO strings and arrays to Postgres array literal strings (`'{1,2}'`) with explicit `::timestamptz` / `::int[]` / `::text[]` casts in the SQL. Built `DashboardContent` client component with `FilterBar`, 4 summary cards with loading skeletons, and `TicketsOverTimeChart` (Recharts `LineChart`, two series). Dashboard page at `src/app/dashboard/page.tsx` renders the client component; both TanStack queries include `filters` in their query keys for automatic refetch on filter change.

## Milestone 3: RLS Policies + Database Role Setup

Created a `rls_user` Postgres role (NOLOGIN, NOBYPASSRLS) and granted it baseline access to the public schema. Enabled RLS with FORCE on all 7 tables. Added 4 Postgres helper functions (`get_app_user_role`, `get_app_client_id`, `get_app_user_id`, `get_app_team_member_id`) that read transaction-local `set_config` variables â€” all without SECURITY DEFINER so they run as the calling role. Created SELECT, INSERT, and UPDATE policies: team members see all data; clients are filtered to their own `client_id`. The `messages_insert` policy enforces attribution â€” team member inserts must match `from_team_member_id = get_app_team_member_id()`, client inserts must have `from_team_member_id IS NULL`, preventing either party from spoofing the other. Policies verified manually in Supabase SQL Editor. Updated `src/lib/db/index.ts` to a single `adminDb` connection (removed `rlsDb` and `DATABASE_RLS_URL`). Created `src/lib/db/rls-client.ts` with `withRLS(ctx, fn)`: opens a transaction on `adminDb`, issues `SET LOCAL ROLE rls_user` as the first statement (transaction-scoped, auto-resets on commit/rollback), then sets all 4 session vars. `withRLS` accepts the full `UserContext` object from `getUserContext()`.
